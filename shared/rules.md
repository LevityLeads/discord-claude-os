# Shared Rules -- All Teams

These rules apply to every team. Read and follow them.

---

## 1. Memory Logging

**At the START of every session:** Read \`memory.md\` in your team directory.

**At the END of every significant action:** Write a log entry to \`memory.md\`:

\`\`\`markdown
## YYYY-MM-DD HH:MM -- [What happened]
- Result: [outcome]
- Next: [follow-up if any]
\`\`\`

If you did something and didn't write it to memory.md, it didn't happen.

## 2. Check Env Vars for API Access

When you need API access:
1. Check environment variables first. Most keys are available as \`$ENV_VAR_NAME\`.
2. Never hardcode API keys. Never commit them.

## 3. Read Context Before Acting

Always read:
- \`context.md\` -- project brief, KPIs, key people
- \`memory.md\` -- recent decisions, learnings, open items

You need this context to act effectively.

## 4. Stay In Your Lane

Each team has boundaries. Don't cross into other teams' domains. Escalate or hand off when something isn't your domain.

## 5. Escalate to General When Needed

Bring in the general assistant when:
- Need context from OTHER projects
- Question is about meta/ops/abilities
- Genuinely unsure what to do

## 6. Be Proactive, Not Just Reactive

Don't wait to be asked. If you see an obvious next step, a problem brewing, or an improvement opportunity, flag it.

## 7. Discord Formatting

**Never use markdown tables in Discord.** They don't render. Use bullet points with bold headers, numbered lists, or code blocks.

**Wrap URLs in \`<>\`** to prevent embeds.

2000 character limit per message. Split long responses.

## 8. No Fabricated URLs

Never make up URLs. Only include links from actual search results, files, or known sources. Zero tolerance.
